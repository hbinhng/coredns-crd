# Performance baseline (2026-04-30)

A first-cut load test of `coredns-crd` against a single-node k3s
cluster. Establishes "what does the plugin do under sustained load?"
without claiming an absolute ceiling — the load generator runs on the
same node, so the numbers are bounded by total box capacity, not by
the plugin's intrinsic throughput.

## Environment

- **Node**: AMD Ryzen 7 5800X, 4 vCPU allocated, 4 GiB RAM, Ubuntu 24.04
- **Cluster**: single-node k3s v1.35.4
- **Replicas**: 2 (leader + follower, sharing load via the kube-dns Service)
- **Resource limits**: requests `cpu=100m memory=70Mi`, limits `memory=200Mi` (chart defaults)
- **Load generator**: `dnsperf 2.10.0`, 4 in-flight clients, query list of fresh A records
- **Co-location penalty**: dnsperf, k3s control plane, and both `kube-dns` pods all on the same 4-vCPU node

## Method

1. `bash test/load/seed.sh <N>` creates N DNSSlices each carrying 50 A records
   (50N records total) with names like `r1.s1.example.com` etc., and writes
   the matching dnsperf query list to `test/load/queries.txt`.
2. `bash test/load/run.sh <duration> [clients]` runs dnsperf for the given
   duration with the given client-concurrency, captures the dnsperf summary,
   and snapshots `kubectl top pod` before and after.

## Results

### Throughput vs. index size (30 s runs, 4 clients)

| Records | QPS | Avg latency | Max latency | Loss |
|---|---|---|---|---|
| 50 | 56,732 | 760 µs | 14 ms | 0% |
| 500 | 57,425 | 789 µs | 12 ms | 0% |
| 5,000 | 54,020 | 838 µs | 16 ms | 0% |
| 25,000 | 57,805 | 902 µs | 28 ms | 0% |

QPS is flat from 50 to 25,000 records. Index lookup is O(1) (hash map
keyed by `(FQDN, type)`); the index size doesn't matter for the hot
path. The slight latency drift (760 µs → 902 µs) is in the noise.

### Soak (120 s, 25,000 records, 4 clients)

| Metric | Start | After 120 s |
|---|---|---|
| QPS | — | 54,888 sustained |
| Loss | — | 0% |
| Pod memory (replica 1) | 100 Mi | 105 Mi |
| Pod memory (replica 2) | 100 Mi | 103 Mi |
| Pod CPU (replica 1) | — | 1183 m |
| Pod CPU (replica 2) | — | 1189 m |

No memory growth indicative of a leak. CPU peaks around 1.2 cores per
replica under load; with `--cpus=4` allocated to the node, both pods +
the load generator + k3s control plane fit. Both pods serving traffic —
the Service load-balances across the two replicas.

### Memory per record

100 Mi at 25,000 records ≈ 4 KB per record (in-memory `dns.RR` struct +
index map overhead + miekg/dns internals). The default 200 Mi limit
holds comfortably up to ~50,000 records; raise the limit beyond that
or add replicas.

## Caveats

- **Co-located load gen.** dnsperf, k3s components, and the load target
  share 4 vCPU. A dedicated load box would push the QPS higher; how
  much higher is unmeasured.
- **Single record type.** All queries were A records. AAAA, TXT, SRV,
  CNAME paths have similar code shapes (same lookup, different RR
  marshal); not separately measured.
- **No conflict storm test.** The conflict-arbitration code path
  (Upsert + diffSiblings) wasn't stressed here; status patch
  throughput under hundreds of concurrent slice updates is a separate
  scenario worth measuring.
- **No multi-node test.** The Helm chart's topology spread + leader
  election were not exercised against a real two-node cluster.

## Recommendations

- **Defaults are appropriate** for clusters up to ~50,000 records and
  moderate query volumes (~50k QPS per replica). No chart changes.
- **At larger scale** (100k+ records, sustained ≥100k QPS), bump
  `resources.limits.memory` to 500 Mi and `replicaCount` to 3+ in
  `values.yaml`. The Service round-robins across replicas naturally.
- **Conflict storm benchmark** is the natural follow-up if production
  telemetry shows status-patch backpressure.

## Reproduction

```bash
# 1. Build dnsperf image (once).
docker build -t dnsperf:local test/load
# Load into your cluster runtime:
kind load docker-image dnsperf:local --name <cluster>     # KinD
docker save dnsperf:local | sudo k3s ctr images import -  # k3s

# 2. Seed.
bash test/load/seed.sh 100   # 100 slices x 50 entries = 5,000 records

# 3. Run.
bash test/load/run.sh 60 4   # 60 seconds, 4 in-flight clients

# 4. Cleanup load slices.
kubectl delete dnsslice -l load=true
```
